01Strategi Pemodelan NoSQL

Enerlyze menggunakan kombinasi strategi Referencing dan Embedding yang dipilih sesuai karakteristik tiap fitur aplikasi:

Referencing (normalisasi) dipakai pada entitas master yang berdiri sendiri dan sering berubah. Koleksi areas_enerlyze menyimpan user_id sebagai referensi ke users_enerlyze (penanggung jawab area), dan data_enerlyze menyimpan area_id serta created_by sebagai referensi. Pendekatan ini dipilih karena satu user dapat memegang banyak area dan satu area memiliki banyak catatan energi lintas bulan — meng-embed data tersebut akan menduplikasi informasi dan menyulitkan proses edit master User/Area.

Embedding (denormalisasi) dipakai pada koleksi ahp_results_enerlyze. Saat fitur Jalankan Analisis AHP dieksekusi, hasil perhitungan — array criteria (bobot tiap kriteria), matriks criteria_matrix, dan ranking (skor prioritas tiap area) — disimpan langsung sebagai array/map di dalam satu dokumen hasil. Hasil AHP bersifat snapshot historis: sekali dihitung untuk satu bulan, nilainya tidak boleh ikut berubah walau master Area atau Kriteria kemudian disunting. Embedding menjaga laporan tetap konsisten dan membuat halaman Laporan Detail AHP cukup membaca satu dokumen.

Contoh konkret: dokumen ahp_results_enerlyze/2026-04 menyimpan ahp_id "AHP-2026-04", objek consistency berisi cr = 0.041, serta array ranking yang sudah memuat area_name "Lantai 1" hasil snapshot — tanpa perlu join ke areas_enerlyze.

Catatan teknis (nested array): Firestore tidak mendukung nested array (array of arrays). Matriks pairwise kriteria 4×4 karena itu tidak disimpan sebagai number[][], melainkan diserialisasi ke bentuk map-of-rows { rows: [{ cells: [...] }] } (helper serializeMatrix() pada src/lib/ahp.js), lalu dideserialisasi kembali ke matriks 2D dengan deserializeMatrix() saat ditampilkan di halaman Laporan Detail AHP.

02Struktur Koleksi

Enam koleksi root pada Firestore. Setiap koleksi diberi sufiks _enerlyze sesuai konvensi proyek.

  • users_enerlyze  — doc ID: auto (mis. u_8Kx2Lp)
    • name string
    • email string
    • password string — format salt:hash (PBKDF2)
    • role string — "admin" | "staff"
    • created_at timestamp
    • updated_at timestamp
  • areas_enerlyze  — doc ID: auto (mis. ar_L1Gf9)
    • name string — mis. "Lantai 1"
    • description string — mis. "Lobby, Food Court"
    • user_id string · ref → users_enerlyze
    • created_at timestamp
    • updated_at timestamp
  • criteria_enerlyze  — doc ID = key (energy, cost, duration, maintenance)
    • key string — identitas stabil kriteria
    • code string — KE / BL / DO / FM
    • name string
    • rank number — 1..4 (unik)
  • data_enerlyze  — doc ID: auto (mis. dt_Apr01)
    • area_id string · ref → areas_enerlyze
    • month string — format YYYY-MM
    • energy number — kWh
    • energy_area_m2 number — luas area (m²)
    • cost number — Rupiah
    • duration number — jam operasional
    • maintenance number — frekuensi/bulan
    • created_by string · ref → users_enerlyze
    • created_at / updated_at timestamp
  • ahp_results_enerlyze  — doc ID = month key (mis. 2026-04)
    • month string · ahp_id string — mis. "AHP-2026-04"
    • criteria array<map> — embedded: {key, code, name, rank, weight}
    • criteria_matrix map — matriks pairwise 4×4 bentuk { rows: [{ cells: [...] }] } (Firestore-safe, bukan nested array)
    • criteria_weights array<number>
    • consistency map — {lambdaMax, ci, cr, isConsistent}
    • ranking array<map> — embedded: {area_id, area_name, area_description, score, breakdown, rank}
    • created_by string · created_at timestamp
  • notifications_enerlyze  — doc ID: auto
    • user_id string · ref → users_enerlyze
    • type string — "missing-data" | "ahp-pending"
    • message string
    • link string — rute tujuan saat notifikasi diklik
    • created_at timestamp

Relasi antar koleksi diwujudkan lewat field referensi ID: areas_enerlyze.user_id → users_enerlyze, data_enerlyze.area_id → areas_enerlyze, data_enerlyze.created_by & notifications_enerlyze.user_id → users_enerlyze.
